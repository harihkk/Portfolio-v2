## Issues being addressed in this pass

### 1. Overused generic reveal

| Field       | Detail                                                                                                                                                                                                                                                                                                                                                      |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| File        | `components/motion/Reveal.tsx`, applied across `app/page.tsx`, `app/systems/page.tsx`, `app/research/page.tsx`, `app/notes/page.tsx`, `app/about/page.tsx`, `app/contact/page.tsx`, `app/experience/page.tsx`, route detail pages, and `components/editorial/*`, `components/evidence/*`, `components/systems/*`                                            |
| Severity    | Medium                                                                                                                                                                                                                                                                                                                                                      |
| Root cause  | A single `Reveal` wrapper is applied to nearly every block on every page. Each instance creates its own ScrollTrigger and `will-change` arming, so the same fade-and-rise plays everywhere. This dilutes the editorial intent and multiplies ScrollTrigger instances and composited layers.                                                                 |
| Planned fix | Reserve `Reveal` for deliberate, high-value sections. Replace blanket usage with plain server-rendered markup where motion adds nothing, reducing the count of live ScrollTriggers and `will-change` layers. Consolidate repeated sibling reveals into a single staggered parent (the `stagger` path already exists) rather than many independent wrappers. |

### 2. SystemsPlate animates `top` and calls setState in a loop

| Field       | Detail                                                                                                                                                                                                                                                                                                                                                                                     |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| File        | `components/systems/SystemsPlate.tsx` (lines 34-49)                                                                                                                                                                                                                                                                                                                                        |
| Severity    | High                                                                                                                                                                                                                                                                                                                                                                                       |
| Root cause  | The timeline tweens the dot's `top` percentage (a layout-triggering property, not a compositor transform) and fires `setActive(i)` from each step's `onComplete`. The timeline has `repeat: -1`, so while the plate is on screen it drives a React re-render on every stage transition indefinitely.                                                                                       |
| Planned fix | Animate `transform: translateY` (or `yPercent`) instead of `top` so the motion stays on the compositor. Drive the lit-stage styling from GSAP/CSS classes rather than React state, or throttle the state updates so the looped timeline does not re-render the component on every step. The existing ScrollTrigger play/pause that stops the loop offscreen is correct and should be kept. |

### 3. EvidenceLine reads layout and calls setState per pointer move, animates `left`/`width`

| Field       | Detail                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| File        | `components/motion/EvidenceLine.tsx` (lines 44-49 and 61-82, plus the inline `style` on lines 114-121 and 159-164)                                                                                                                                                                                                                                                                                                                |
| Severity    | High                                                                                                                                                                                                                                                                                                                                                                                                                              |
| Root cause  | On every `pointermove`, `pointerToPct` calls `getBoundingClientRect` (a layout read) and the handler then calls `setPct` and `setActive`, triggering a React render per move event. The lit portion animates `width` and the node animates `left`, both layout-affecting properties rather than transforms. The combination of read, state churn, and non-composited animation causes jank during the signature drag interaction. |
| Planned fix | Cache the track rect on pointer down (and on resize) instead of reading it every move. During an active drag, write the node position directly to the DOM (transform-based) and coalesce updates with `requestAnimationFrame`, committing the discrete anchor index to state only on release. Move the node to `transform: translateX` and the lit line to `transform: scaleX` so the drag runs on the compositor.                |

### 4. ContamCheckr bar animates width

| Field       | Detail                                                                                                                                                                                                                            |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| File        | `components/systems/demos/ContamCheckrDemo.tsx` (lines 55-68, bar element line 139)                                                                                                                                               |
| Severity    | Low                                                                                                                                                                                                                               |
| Root cause  | The evidence bar animates `width` via an inline CSS transition each time the index changes. `width` is a layout-triggering property, so the transition is not compositor-only. The reduced-motion branch is already handled.      |
| Planned fix | Animate `transform: scaleX` with `transform-origin: left` instead of `width`, keeping the same eased timing and the reduced-motion instant-set branch. The dashed marker positioned by `left` is static per state and can remain. |

### 5. Header setState on every scroll

| Field       | Detail                                                                                                                                                                                                                                                                               |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| File        | `components/layout/SiteHeader.tsx` (lines 16-21)                                                                                                                                                                                                                                     |
| Severity    | Medium                                                                                                                                                                                                                                                                               |
| Root cause  | The scroll handler calls `setStuck(window.scrollY > 24)` on every scroll event. Even though React bails out when the boolean is unchanged, the handler runs the comparison and schedules work on every scroll frame, and the listener is not coalesced with `requestAnimationFrame`. |
| Planned fix | Only call `setStuck` when the boolean actually flips (guard against the current value), and coalesce the scroll handler with `requestAnimationFrame` as `ReadingProgress.tsx` already does. The listener is already `{ passive: true }`, which is correct and should stay.           |

### 6. Persistent will-change

| Field       | Detail                                                                                                                                                                                                                                                                                                                                                           |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| File        | `app/globals.css` (line 251, rule `:root[data-reveal-ready] [data-reveal]`) and `components/motion/Marquee.tsx` (line 76, `will-change-transform`)                                                                                                                                                                                                               |
| Severity    | Medium                                                                                                                                                                                                                                                                                                                                                           |
| Root cause  | `will-change: transform, opacity` is applied to every `[data-reveal]` element for the whole session once `data-reveal-ready` is set, and the marquee track carries `will-change-transform` permanently. `will-change` left on long-lived elements forces the browser to hold extra compositor layers and memory even after the animation has finished.           |
| Planned fix | Apply `will-change` only for the duration of the reveal animation (set it when the tween starts, clear it on completion via GSAP callbacks) rather than as a static CSS rule on all reveal wrappers. For the marquee, keep `will-change-transform` only while the loop is running. This pairs naturally with fix 1, which reduces the number of reveal wrappers. |

### 7. Reveal could hide content

| Field       | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| File        | `components/motion/Reveal.tsx` (lines 41-67), `app/globals.css` (lines 249-252), `app/layout.tsx` (line 77 inline script)                                                                                                                                                                                                                                                                                                                                  |
| Severity    | High                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| Root cause  | The CSS rule sets `opacity: 0` on `[data-reveal]` as soon as `data-reveal-ready` is on the document. Visibility then depends on the GSAP tween firing. If a ScrollTrigger fails to initialize, mis-measures start position, or an error interrupts the effect, an armed element can stay at `opacity: 0`, hiding real content. The no-JS and reduced-motion paths are safe (content stays visible), but the JS-enabled path has a single point of failure. |
| Planned fix | Add a safety net so content cannot stay hidden: a fallback timeout or `ScrollTrigger.refresh` verification that forces `opacity: 1` if a reveal has not played within a short budget, and ensure the `data-reveal-ready` arming is only applied to elements that have a confirmed live trigger. Keep the existing reduced-motion override (line 267-269) that restores `opacity: 1`.                                                                       |

### 8. background-attachment: fixed

| Field       | Detail                                                                                                                                                                                                                                                                                             |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| File        | `app/globals.css` (line 104, on the page background gradient layers)                                                                                                                                                                                                                               |
| Severity    | Medium                                                                                                                                                                                                                                                                                             |
| Root cause  | `background-attachment: fixed` forces the browser to repaint the fixed background against the viewport on every scroll, which is a known scroll-performance and jank source, especially on mobile and high-DPI displays, and compounds with the other scroll-time work (header, reading progress). |
| Planned fix | Remove `background-attachment: fixed` and achieve the same static-gradient look with a normally attached background, or move the decorative gradient to a fixed-position pseudo-element layer that is composited rather than repainted on scroll.                                                  |

## Summary of severities

| Severity | Issues                                                                                                                                     |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| High     | SystemsPlate `top` + looped setState (2); EvidenceLine layout-read + per-move setState + `left`/`width` (3); reveal could hide content (7) |
| Medium   | Overused generic reveal (1); header setState on every scroll (5); persistent will-change (6); `background-attachment: fixed` (8)           |
| Low      | ContamCheckr bar width animation (4)                                                                                                       |

The common thread is favoring compositor-only properties (`transform`, `opacity`) over layout-triggering ones (`top`, `left`, `width`, fixed background), coalescing scroll and pointer work with `requestAnimationFrame`, scoping `will-change` to active animations, and guaranteeing content visibility on the JS path as the no-JS and reduced-motion paths already do.
